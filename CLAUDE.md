# CLAUDE.md — このリポジトリでの開発心得

小学生向け教育アプリ集（作者: kagasen先生）。GitHub Pages で公開する静的サイトで、
各アプリ＝「フォルダ + index.html」、ポータル＝ルートの `index.html` + `apps.js`。
このファイルは、これまでの開発で確立した「壊してはいけない前提」と「品質を保つ習慣」の集大成。
**新しいアプリを作るとき・既存アプリを触るときは、まずここを読み、次にそのアプリの `HANDOFF.md` を読むこと。**

---

## 1. 崩さない大前提（全アプリ共通）

- **完全オフライン**: サーバー・API・DB・CDN読み込みなし。フォントもシステムフォント（オフライン初回起動でも崩れない）。外部ライブラリが必要なら同梱する（例: kanji-bouken の three.min.js）。
- **単一 index.html・ビルドなし**: CSS/JS込みの1ファイル。React+Vite等のビルド構成は過去に提案して**ユーザー了承のうえ不採用**になった。勝手に技術スタックを変えない（例外: typing だけ esbuild を使う）。
- **保存は localStorage のみ**: 端末ごと・同期なし・1端末1人。**子どもの行動データを外部に送信しない**。これは無料運用と子どものプライバシー保護のための意図的な設計。
- **APIキーが要る機能は不採用**: AIコーチ等はこの理由で見送った前例あり。代替はルールベースの大量プール（アドバイス文など）。
- **通知は Plan A**: OS通知なし。PWA化＋起動時の状態依存あいさつで代替（定時Web通知はバックエンド必須で前提が崩れるため）。
- **PWA構成**: `manifest.json` / `sw.js` / `icon.svg` を各アプリに置く。
- 「iPad/スマホ両対応」＝**レスポンシブ**の意味。ネイティブアプリ化ではない。

## 2. 作業の進め方

1. **実装前に必ずそのアプリの `HANDOFF.md` を読む**。これが設計合意と実装状況の source of truth。新アプリでは最初に作る。
2. **設計合意ファースト**: 大きな機能は作る前に前提・仕様を質問で詰めて合意する（/grill-me の流儀）。ユーザーが明示的に決めたこと（例: とびばこ追加、チートコード温存）は勝手に変えない。
3. **動く骨格ファースト**: まず最小限で動くものを作り、段階的に肉付けする（第1段→第2段→第3段）。数値バランス（XP・価格・必要レベル等）は全部「叩き台」で実機調整前提と明記する。
4. **既存コードのスタイルに合わせる**: 命名・コメント密度・ひらがな変数名やUI文言のトーンまで、そのファイルの流儀に従う。
5. **完了報告の前にブラウザで実際に動かして検証する**。「たぶん動く」で報告しない。未検証・失敗はそのまま正直に伝える。
6. **作業後は `HANDOFF.md` を更新**して、次のセッションが読めば続きから作れる状態にする。

## 3. データを絶対に壊さない（最重要）

子どもの記録（勉強・運動・獲得バッジ）は積み重ねそのもの。消えると一番悲しい。

- **スキーマ変更時は `load()` 内で旧データを自動移行する**。前例: すいえい旧12段階のidを新idへ移行、バッジ旧31種のidを新配列に温存。旧データを黙って捨てない。
- **セーブキーの名前空間分離**: 機能・学年を増やしても既存キーと衝突しない設計にする（例: kanji-bouken の `clearedKey`、コードポイントキー）。
- **取り消し操作は付与と対称に**: チェックを外したらXPも巻き戻す。トグルの往復で増減ゼロ、初回ボーナスの二重取り（farming）不可、を必ず確認する。
- **リセットが必要でも学習記録は残す**: 前例 = kanji-bouken の `resetProgressOnce()` はコイン・購入品だけ戻し、図鑑・習得・クリア★は残した。実行済みフラグ（`RESET_FLAG` の日付）で端末ごと1回だけ走る方式。
- **運用レバー・裏技は意図的な仕様**: チートコード（`boukenmappu`）や RESET_FLAG はユーザー確認済みで残している。消さない。

## 4. キャッシュ更新の儀式（忘れると子どもの端末に反映されない）

- データ用 `.js` を更新したら → `index.html` 内の該当 `?v=N` を**必ず繰り上げる**（`?cb=` は index.html しかバストしない）。
- PWAのアセットを更新したら → `sw.js` の `CACHE` バージョンを繰り上げる（v1→v2…）。
- データ生成スクリプト（`build-*.mjs`）があるアプリは、生成物を手編集せず**スクリプトを直して再生成**する。ユーザー提供の字リスト等は**件数・重複チェック必須**（員重複→院、永欠落、の前例あり）。

## 5. 子ども向けUIの流儀

- **表記はひらがな中心**（対象学年で習う漢字まで）。タップターゲット **48px以上**。
- **褒める演出を惜しまない**: レベルアップ、バッジ獲得モーダル、自前コンフェッティ（`confettiBurst()` の前例）。失敗を責める文言は書かない。
- **成長は単調増加**: レベルや達成率がサボりで「下がる」設計にしない（見た目の変化＝ゾンビ化はOK、数値は戻さない）。
- **ガチャ・課金的な運要素は不採用**: レア解放はマイルストーン確定式。対戦の運要素も「大差は実力勝ち・接戦だけ逆転あり」のように理不尽にしない。
- 技・種目には**SVGイラスト**（棒人間＋動き矢印など）を添えて、字が読めなくても分かるようにする。

## 7. 品質を保つための細かい習慣

- 日付またぎ・連続記録（streak）は「今日の日付での記録のみカウント」のように**判定基準を1つに決めて明記**する。
- 定数データ（技・バッジ・アイテム）は**配列1箇所に集約**し、描画は共通関数で回す（`BALL_SPORTS`+`viewBallSport()`、`BADGES`+`test(S)` の前例）。同種の機能を増やすときは「配列に1行足すだけで増える」設計を目指す（kanji-bouken の学年増設で実証済み）。
- 後方互換を持つ入出力（対戦コード等）は旧フォーマットも読めるようにしてから新フォーマットへ移行する。
- 分からないことは推測で書かず、既存アプリの同種実装を読んでから書く。
