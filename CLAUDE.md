# CLAUDE.md — かがせんのHAPPYアプリ集 開発心得

「かがせんのHAPPYアプリ集」（作者: kagasen先生）。**小学生と先生のための教育アプリ集**で、
GitHub Pages で公開する静的サイト。各アプリ＝「フォルダ + index.html」、ポータル＝ルートの `index.html` + `apps.js`。
このファイルは全アプリ共通の「壊してはいけない前提」と「品質を保つ習慣」。
**新しいアプリを作るとき・既存アプリを触るときは、まずここを読み、次にそのアプリの `HANDOFF.md` を読むこと。**
アプリ固有の仕様・経緯・裏技は CLAUDE.md ではなく各アプリの `HANDOFF.md` に書く。

---

## 0. アプリ集の全体像 — 2系統ある

作る・直すときは、まず「どちらの系統のアプリか」を意識する。**新しいアプリを既存アプリの真似で作らない**こと（例: 学習ゲームの流儀を先生用ツールに持ち込まない、逆も同じ）。

- **子ども用（学習・記録系）**: 子どもが自分の端末で毎日使う。例: 漢字・ひらがな・カタカナ練習、タイピング、しりとり、毎日問題、自分レベルアップアドベンチャー、うごきのきろく。localStorage に個人の記録を持ち、続けたくなる仕掛け（§6）を持つものが多い。
- **先生用（授業ツール系）**: 先生が教室の大型モニタや iPad で使う。例: 席替え、グループ分け、給食くじ、学級ボード、体育の作戦ボード・リーグ・トーナメント、思考ツール。**準備なしでその場ですぐ使える**ことが命。個人の蓄積データは基本持たない（名簿等の設定だけ localStorage）。

どちらも共通するのは「**小学生が読める・見て分かる**」こと。先生用ツールでも画面は子どもが見る前提で作る。

## 1. 崩さない大前提（全アプリ共通）

- **完全オフライン**: サーバー・API・DB・CDN読み込みなし。フォントは基本システムフォント（任意の文字を入力するアプリはサブセット同梱不可のため必須）。決まった文字しか表示しないアプリはサブセットフォント同梱も可（`build-fonts.mjs` の流儀）。外部ライブラリが必要なら同梱する（`vendor/` や three.min.js の前例）。
- **単一 index.html・ビルドなし**: CSS/JS込みの1ファイル。React+Vite等のビルド構成は過去に提案して**ユーザー了承のうえ不採用**になった。勝手に技術スタックを変えない（例外: typing だけ esbuild を使う）。
- **保存は localStorage のみ**: 端末ごと・同期なし。**子どもの行動データを外部に送信しない**。これは無料運用と子どものプライバシー保護のための意図的な設計。
- **APIキーが要る機能は不採用**: AIコーチ等はこの理由で見送った前例あり。代替はルールベースの大量プール（アドバイス文など）。
- **通知は Plan A**: OS通知なし。PWA化＋起動時の状態依存あいさつで代替（定時Web通知はバックエンド必須で前提が崩れるため）。
- **PWA構成**: `manifest.json` / `sw.js` / `icon.svg` を各アプリに置く。
- **iPad対応は必須**: 教室でも家庭でも主要端末は iPad。「iPad/スマホ両対応」＝**レスポンシブ**の意味（ネイティブアプリ化ではない）。横向き・縦向き両方で確認する。

## 2. 作業の進め方

1. **実装前に必ずそのアプリの `HANDOFF.md` を読む**。これが設計合意と実装状況の source of truth。新アプリでは最初に作る。
2. **設計合意ファースト**: 大きな機能は作る前に前提・仕様を質問で詰めて合意する（/grill-me の流儀）。ユーザーが明示的に決めたことは勝手に変えない・勝手に消さない。
3. **動く骨格ファースト**: まず最小限で動くものを作り、段階的に肉付けする（第1段→第2段→第3段）。数値バランス（XP・価格・出題数等）は全部「叩き台」で実機調整前提と明記する。
4. **既存コードのスタイルに合わせる**: 命名・コメント密度・ひらがな変数名やUI文言のトーンまで、そのファイルの流儀に従う。
5. **完了報告の前にブラウザで実際に動かして検証する**。「たぶん動く」で報告しない。未検証・失敗はそのまま正直に伝える。
6. **作業後は `HANDOFF.md` を更新**して、次のセッションが読めば続きから作れる状態にする。

## 3. データを絶対に壊さない（記録を持つアプリで最重要）

子どもの記録（勉強・運動・獲得バッジ等）は積み重ねそのもの。消えると一番悲しい。

- **スキーマ変更時は `load()` 内で旧データを自動移行する**。旧データを黙って捨てない（旧id→新idへの移行、旧配列の温存、の前例が複数アプリにある）。
- **セーブキーの名前空間分離**: 機能・学年・プロフィールを増やしても既存キーと衝突しない設計にする。
- **取り消し操作は付与と対称に**: チェックを外したらXPも巻き戻す。トグルの往復で増減ゼロ、初回ボーナスの二重取り（farming）不可、を必ず確認する。
- **リセットが必要でも学習記録は残す**: 通貨・購入品だけ戻し、図鑑・習得・クリア記録は残す方式が前例。実行済みフラグで端末ごと1回だけ走らせる。
- **運用レバー・裏技は意図的な仕様**: ユーザー確認済みで残しているチートコードやリセットフラグが存在する。詳細は各アプリの `HANDOFF.md` に記録し、勝手に消さない。
- 先生用ツールでも、先生が打ち込んだ名簿・設定は同じ扱い（黙って消えない・移行する）。

## 4. キャッシュ更新の儀式（忘れると子どもの端末に反映されない）

- データ用 `.js` を更新したら → `index.html` 内の該当 `?v=N` を**必ず繰り上げる**（`?cb=` は index.html しかバストしない）。
- PWAのアセットを更新したら → `sw.js` の `CACHE` バージョンを繰り上げる（v1→v2…）。
- データ生成スクリプト（`build-*.mjs`）があるアプリは、生成物を手編集せず**スクリプトを直して再生成**する。ユーザー提供の字リスト等は**件数・重複チェック必須**（重複・欠落の前例あり）。

## 5. 小学生向けUIの流儀（全アプリ共通）

- **表記はひらがな中心**（対象学年で習う漢字まで）。先生だけが触る設定画面は普通の日本語でよいが、子どもが見る画面は必ずひらがな基準。
- タップターゲット **48px以上**。iPadの指タッチで押しやすく。
- **教室の大型モニタに映しても見える**文字サイズ・コントラスト（先生用ツールはとくに）。
- 失敗を責める文言は書かない。「ざんねん」より「もういちど やってみよう！」。
- 技・種目・選択肢には**SVGイラスト**（棒人間＋動き矢印など）を添えて、字が読めなくても分かるようにする。

## 6. ゲーミフィケーションの流儀（報酬・記録・成長要素があるアプリのみ）

学習・記録系アプリに続けたくなる仕掛けを入れるときのルール。先生用ツールには適用しない。

- **褒める演出を惜しまない**: レベルアップ、バッジ獲得モーダル、自前コンフェッティ（`confettiBurst()` の前例）。
- **成長は単調増加**: レベルや達成率がサボりで「下がる」設計にしない（見た目の変化＝ゾンビ化はOK、数値は戻さない）。
- **ガチャ・課金的な運要素は不採用**: レア解放はマイルストーン確定式。対戦の運要素も「大差は実力勝ち・接戦だけ逆転あり」のように理不尽にしない。
  ※席替え・給食くじ等の「公平なランダムが目的のツール」はこの話と別物。
- 連続記録（streak）・日付またぎは「今日の日付での記録のみカウント」のように**判定基準を1つに決めて明記**する。

## 7. 公開時チェックリスト

- [ ] ルートの `apps.js` にカード追加、`images/` にサムネイルSVG追加
- [ ] `manifest.json` / `sw.js` / `icon.svg` を用意（PWA）
- [ ] **出典・ライセンスクレジットを表記**し、以後絶対に消さない（例: KanjiVG © Ulrich Apel / CC BY-SA 3.0 は同一ライセンス継承必須。体育の級表は栃木県「みんなが使えるチャレンジカード集」出典）
- [ ] iPad・スマホ両方の画面幅で表示確認
- [ ] オフライン（機内モード）で初回以降起動できるか確認
- [ ] `node release-check.mjs` で外部読み込み等の警告がないか確認
- [ ] `HANDOFF.md` を最新化
- [ ] コミットメッセージは既存流儀（`feat: 日本語で内容`）

## 8. 品質を保つための細かい習慣

- 定数データ（技・バッジ・アイテム・お題）は**配列1箇所に集約**し、描画は共通関数で回す。同種の機能を増やすときは「配列に1行足すだけで増える」設計を目指す（複数アプリで実証済み）。
- 後方互換を持つ入出力（対戦コード・バックアップ封筒等）は旧フォーマットも読めるようにしてから新フォーマットへ移行する。
- 分からないことは推測で書かず、既存アプリの同種実装を読んでから書く。
